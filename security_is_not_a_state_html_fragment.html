Security Is Not a State

One of the first security exercises I remember doing was almost insultingly simple.

I had a FreeBSD system. I connected to it with Telnet while capturing the traffic in Wireshark. Then I did the same thing with SSH.

The lesson was obvious.

With Telnet, I could inspect the traffic and see what I was doing. Credentials, commands, session data: there it was.

With SSH, I could still see that communication was happening. I could see the endpoints. I could see timing, packet sizes, and the existence of a session. But I could not simply read the application data out of the capture.

At the time, the lesson was:

Telnet is insecure. SSH is secure.

That is not wrong exactly.

It is just a dangerously incomplete way to think.

Years later, I think the more useful lesson was this:

SSH changed what an observer could know.

That is a much narrower statement.

It is also much more useful.

The Problem With Declaring Something Secure

A lot of security thinking starts by defining a state.

We turn off Telnet and enable SSH.

Now we are secure.

We add TLS.

Now we are secure.

We require MFA.

Now we are secure.

We move something behind a VPN.

Now we are secure.

The problem is not that these controls are useless. They are usually real improvements.

The problem is the leap from:

This control prevents a known class of failure.

to:

This system is now secure.

Those are not the same claim.

Using SSH instead of Telnet is entirely defensible. It removes the absurd property where a passive observer can simply read the session contents off the wire.

But the following statements are not equivalent:

The traffic is encrypted.

That does not mean:

The endpoint is trustworthy.

That does not mean:

The implementation has no vulnerabilities.

That does not mean:

The credentials are strong.

That does not mean:

The server is configured correctly.

That does not mean:

The user actually verified the server identity.

That does not mean:

The whole system is secure.

Each statement is stronger than the one before it.

The failure happens when a system, an organization, or sometimes an entire industry silently promotes a weaker statement into a stronger one.

The packets are encrypted, therefore the connection is secure.

The object decoded, therefore the object is valid.

The object is valid, therefore the message is valid.

The message is valid, therefore it is valid at this point in the protocol.

The state is valid, therefore it must have been reached through a valid history.

That last jump is where things get especially interesting.

A Valid State Does Not Prove a Valid History

Consider a protocol as a state-transition system.

You have some state Sn, and a set of rules describing which transitions are allowed.

Given the current state, there are two very different questions:

What states can legally come next?

and:

What previous states could have produced this one?

Those questions are not symmetrical.

A current state may tightly constrain the future while revealing almost nothing about the past.

Or many different histories may converge on exactly the same current state, even though those histories should produce different decisions about what happens next.

That means a system can arrive at a state that is perfectly valid in isolation but insufficient for making the next correct decision.

Here is a nontechnical example.

Imagine a group of parents who carpool their children.

Sometimes accidents happen during the ride, so the group has some rule for buying extra diapers. Everyone wants to contribute roughly their fair share. Nobody is trying to cheat anyone. In fact, some people may deliberately contribute a little extra.

The group keeps track of only one thing:

There is currently $40 in the diaper fund.

That sounds like state.

But how did the fund reach $40?

Maybe Alice paid $20 and Bob paid $20 while Carol paid nothing.

Maybe Carol paid the entire $40.

Maybe everyone contributed different amounts over several trips.

Maybe one parent already covered several emergency purchases directly and then also contributed to the fund.

All of those histories can converge on:

Current balance: $40.

But the correct future contribution is different depending on the history.

If the protocol has discarded that history and preserved only the current balance, then the current state is not sufficient for making the next fair decision.

Everyone can be honest.

Everyone can follow the rules.

Everyone can try to contribute fairly.

And the system can still fail.

Not because a participant was malicious.

Because the protocol destroyed information it later needed.

A useful way to state the problem is:

A state is sufficient only when the histories that can produce that state do not require meaningfully different future decisions.

Once distinguishable histories collapse into an indistinguishable present, the system may no longer know enough to behave correctly.

What Exactly Do You Know?

This is the question I keep coming back to.

Not:

Is this secure?

But:

What do I actually know?

And then:

What am I inferring?

And finally:

What does the protocol actually prove?

That way of thinking naturally creates a stack.

bytes
↓
decoded object
↓
structurally valid object
↓
semantically valid object
↓
valid message
↓
valid message at this point in the protocol
↓
valid state transition
↓
valid reachable state
↓
sufficient history for the next decision

Success at one layer does not automatically prove the next.

The bytes decoded.

Wonderful.

That tells you the bytes decoded.

The object matches the expected schema.

Wonderful.

That tells you the object matches the expected schema.

The message contains a legal value for every field.

Wonderful.

That does not prove the sender was allowed to send that message now.

That is probably why this line of thought makes me think about things like ASN.1.

A representation can decode correctly.

A decoded object can satisfy its structural constraints.

That still does not tell you whether the message is semantically appropriate, whether the sender was authorized to produce it, or whether it could have been reached through the legitimate protocol history you think preceded it.

Those are separate questions.

Security failures often happen because someone asks the first question and believes they answered the last one.

The Web 2.0 Mistake

This also gets at one of the mistakes I associate with a lot of Web 2.0-era application design.

A server hands something to a client.

Later, the client hands something back.

The application behaves as though the returned state has continuity with the state it originally sent.

But what exactly does the server know?

It knows that it sent an object.

It knows that later it received an object.

Those facts do not, by themselves, establish that the second object is the first object returning home unchanged.

That sounds obvious when stated plainly, but applications have repeatedly made some version of this mistake.

Hidden fields.

Client-maintained values.

Serialized state.

Unsigned objects.

UI restrictions treated as authority boundaries.

Object identifiers that were never expected to be modified because the application itself never offered a button for modifying them.

The mistake is deeper than “validate your inputs.”

Input validation matters, obviously.

But a state can be completely valid and still be illegitimate.

Suppose a protocol permits:

A → B → C → D

The server gives the client state B.

Later, the client presents D.

D may be a valid state.

The question is whether the system can establish that the permitted transition path occurred.

Now imagine:

      B → C → D
     /
A ──
     \
      X ───→ D

Seeing D tells you even less.

The current state may be valid.

It may also be reachable from multiple histories with very different meanings.

A valid state does not prove a valid history.

And if future decisions depend on knowing that history, then the protocol has a problem.

Security Theater Begins at the End of the Question

This is where I think security theater actually begins.

Not when someone deploys a security control.

Deploying SSH instead of Telnet is not theater.

Deploying TLS is not theater.

Using MFA is not theater.

Security theater begins when the existence of the control is used to stop further inquiry.

We use SSH.

Okay.

What threat are we discussing?

The traffic is encrypted.

Okay.

Against whom?

We require MFA.

Okay.

What happens after authentication?

The service is behind a VPN.

Okay.

What do you trust about the client that has connected?

A real security improvement says:

We identified a threat and introduced a control that changes the attacker’s capabilities.

Security theater says:

We performed an action associated with security and have therefore entered the Secure State.

There is no Secure State.

There are properties.

There are guarantees.

There are assumptions.

There are costs imposed on attackers.

There are things the system knows and things it merely believes.

The job is to keep those categories separate.

Back to Wireshark

That old Wireshark exercise was useful because it made one security property visible.

With Telnet, the observer could see the session contents.

With SSH, the observer could not trivially recover them from the packet capture.

That is a real change.

It is measurable.

It is useful.

It is defensible.

But the mature lesson is not:

SSH is secure.

The mature lesson is:

SSH changes the knowledge available to a particular observer under a particular threat model.

That sentence is less satisfying.

It does not fit on a compliance checkbox.

It does not give management a binary answer.

It is also much closer to how security actually works.

The Weird Box in the Corner

Looking back, a lot of my career has involved this same question in different forms.

I have often been the person sent to look at the weird box or binary in the corner that nobody quite understands.

An old Unix machine.

A Windows 98 system everyone believes is critical.

A piece of hospital infrastructure nobody wants to touch.

A malware sample.

The useful question is rarely:

What is this supposed to be?

It is:

What does it actually do?

What communicates with it?

What state does it maintain?

What assumptions do other systems make about it?

What happens if it disappears?

What behavior must be preserved?

What history matters?

What can be inferred from the current state, and what has already been lost?

A machine can become sacred because someone once said, “Do not touch this box,” and then disappeared.

Years later, everyone remembers the warning but nobody remembers the reason.

Eventually the organization is not protecting a known dependency.

It is protecting a myth.

Sometimes, once the system is actually understood, the requirement turns out to be much smaller:

This executable needs to keep working at this IP address.

That is a very different problem from:

This exact physical machine must exist forever.

Again, the useful work begins by separating what is known from what is assumed.

The Question I Wish More Systems Asked

Security is full of declarative statements.

Encrypted.

Authenticated.

Authorized.

Validated.

Trusted.

Secure.

The problem is that every one of those words is incomplete without context.

So I think the more useful habit is to keep asking:

What do I know?

What am I inferring?

What does the system actually prove?

And, when dealing with protocols:

Given this state, what can I determine about the future?

Given this state, what can I determine about the past?

Has the system preserved enough history to make the next decision correctly?

Because a parser can succeed while the application fails.

A message can be valid while the transition is invalid.

A state can be valid while the history is impossible.

A security control can work while the system remains vulnerable.

And a system can be more secure without ever becoming secure.

That is not a reason to stop improving it.

It is a reason to be precise about what each improvement actually gives you.
